# Security Review Report
- Project: 2026-game
- Stack: Go chi API + Next.js
- Context: system lifetime = 1 day
- Reviewed at: 2026-03-23_16-59-26

## Executive Summary
- The highest-value issues are all around token handling, not classic injection: privileged login tokens are accepted via URL query string, kept in browser-visible storage, and in one case already shown leaking into request traces.
- This matters even for a 1-day system because admin/staff/booth links will be actively shared on phones and laptops during the event; leaked links or browser-visible tokens can be replayed immediately for admin actions, coupon issuance, or booth check-ins.
- A second issue is build-time secret exposure: backend Docker build copies the whole backend context, so a local `.env` can be baked into remote build context/layers.
- I did not find a concrete authz bypass, SQL injection, or obvious XSS sink worth prioritizing over the above.

## Findings
### [High] Privileged tokens are passed in URL query strings and can leak through logs/history
- Where: `frontend/app/login/page.tsx`, `frontend/app/(internal)/admin/page.tsx`, `frontend/app/(internal)/booth/page.tsx`, `frontend/app/(internal)/cashier/page.tsx`, `frontend/components/ui/ManualTokenInput.tsx`, `frontend/.next/dev/trace:13`
- Why it matters: `?token=` secrets are exposed to browser history, screenshots, copied links, referrers/internal logs, and can be replayed immediately to obtain player/admin/staff/booth sessions.
- Evidence: pages read `searchParams.get("token")` and log in with it; `ManualTokenInput` redirects to `/login?token=...`; local Next trace already contains full admin URLs with the token value.
- Fix: stop accepting login secrets via query string; accept only POST body or `Authorization` header, then immediately `replaceState`/redirect to a clean URL; rotate any issued links/tokens already shared this way.
- Confidence: High

### [High] Frontend stores auth and operator tokens in JS-readable storage
- Where: `frontend/stores/userStore.ts`, `frontend/stores/staffStore.ts`, `frontend/stores/boothStore.ts`, `frontend/hooks/api/useUser.ts`
- Why it matters: any same-origin XSS, malicious extension, or third-party script can steal player/staff/booth tokens; for player auth this also defeats the backend's HttpOnly cookie protection.
- Evidence: player token is persisted by Zustand and written with `document.cookie`; staff/booth raw login tokens are persisted in localStorage-backed stores even after server session cookies are issued.
- Fix: do not persist raw tokens in Zustand/localStorage/document.cookie; keep only minimal UI state client-side and rely on server-set HttpOnly cookies for session continuity.
- Confidence: High

### [Medium] Backend Docker build can ingest local `.env` secrets into build context
- Where: `backend/Dockerfile:13`, `backend/.gitignore:1`
- Why it matters: if remote builders/cache are used, local secrets can leave the workstation and persist in build layers or logs.
- Evidence: backend ignores `.env` in git, but Docker build does `COPY . .`; no `backend/.dockerignore` is present, so local `.env` is included in build context by default.
- Fix: add `backend/.dockerignore` excluding `.env`, `tmp`, local data, and other non-build files; prefer explicit `COPY` of only required source files.
- Confidence: High

### [Medium] `/metrics` is exposed without auth or environment gating
- Where: `backend/cmd/api/main.go:145`
- Why it matters: public metrics can disclose internal routes, traffic patterns, runtime details, and operational state that help live abuse during the event.
- Evidence: `r.Handle("/metrics", promhttp.Handler())` is mounted outside `/api` with no middleware or env check.
- Fix: disable in public deployments, or restrict by network/auth proxy.
- Confidence: High
