# Security Review Report
- Project: 2026-game
- Stack: Go chi API + Next.js
- Context: system lifetime = 1 day
- Reviewed at: 2026-03-23_16-55-52

## Executive Summary
- The highest-value fixes are session/token handling and game-flow abuse, not generic hardening.
- The frontend currently defeats the backend's `HttpOnly` session model by copying player tokens into a JS-readable cookie and persisted storage; one token leak is enough for same-day account takeover.
- Privileged login links (`admin` / `cashier` / `booth`) are URL-token based, and admin sessions reuse the raw global `ADMIN_KEY`; this makes leaked links/cookies far more damaging than they need to be.
- `/games/submissions` can advance levels without any gameplay proof, so a scripted client can farm progression/coupons/ranking inside the event window.

## Findings
### [High] Frontend downgrades player session to JS-readable storage
- Where: `frontend/stores/userStore.ts:19`, `frontend/hooks/api/useUser.ts:59`, `frontend/components/providers/AuthGuard.tsx:33`
- Why it matters: a stolen browser token is enough to act as the player for the rest of the event; this defeats the backend's `HttpOnly` cookie protection.
- Evidence: after login, the app writes `token` with `document.cookie` and persists `authToken` via Zustand `persist`; subsequent auth checks rely on that client-side token state.
- Fix: stop writing the auth token from JS; rely only on the backend-set `HttpOnly` cookie, or keep any client token in memory only.
- Confidence: High

### [High] Game progression can be scripted without proving gameplay
- Where: `backend/internal/handler/game/submit.go:24`, `backend/internal/router/game.go:23`
- Why it matters: this directly affects rankings, coupon issuance, and event fairness within 1 day.
- Evidence: `POST /api/games/submissions` increments `current_level` after only checking `unlock_level` and elapsed time since `LastPassTime`; no score, signature, replay nonce, or server-verifiable game result is required.
- Fix: require a server-verifiable completion artifact before level-up, or temporarily disable coupon/leaderboard advancement from this endpoint.
- Confidence: High

### [High] Admin session cookie is the raw global admin secret
- Where: `backend/internal/handler/admin/login.go:14`, `backend/internal/handler/admin/login.go:38`, `backend/pkg/middleware/admin_key.go:15`
- Why it matters: any leaked `admin_token` cookie is not just a session - it is the master admin credential, reusable until the secret is rotated.
- Evidence: admin login compares `Authorization` directly to `config.Env().AdminKey`, then sets `admin_token` to that same value for `30 * 24 * time.Hour`; admin middleware later accepts the cookie only by equality with `AdminKey`.
- Fix: use `ADMIN_KEY` only as bootstrap auth, then issue a separate short-lived signed session token/session ID.
- Confidence: High

### [Medium] Privileged tokens are passed in URL query strings and left in browser-visible state
- Where: `frontend/app/(internal)/admin/page.tsx:13`, `frontend/app/(internal)/cashier/page.tsx:85`, `frontend/app/(internal)/booth/page.tsx:27`, `frontend/app/login/page.tsx:69`, `frontend/components/ui/ManualTokenInput.tsx:46`
- Why it matters: URLs get copied, screenshotted, logged, stored in history, and exposed on shared devices; for staff/admin/booth flows this can become immediate privilege misuse.
- Evidence: the app reads `?token=` from the URL for player/admin/staff/booth login, does not scrub it first, and the player error page even renders the raw token back to the UI.
- Fix: replace URL-token login with POST or one-time code exchange; at minimum strip `token` from the address bar immediately before rendering anything.
- Confidence: High

### [Medium] Unauthenticated login/bootstrap endpoints are not rate-limited
- Where: `backend/internal/router/admin.go:18`, `backend/internal/router/user.go:20`, `backend/internal/router/discount.go:20`, `backend/internal/router/activities.go:20`, `backend/pkg/middleware/session_rate_limit.go:28`
- Why it matters: these are the few endpoints an external attacker can hammer directly during the event; if any token space is weak or leaked partially, there is no throttle.
- Evidence: `sessionRateLimit` only works after auth context exists; all `/session` routes are mounted before auth middleware and receive no separate IP-based limit.
- Fix: add an IP-based limiter/backoff on unauthenticated session creation endpoints.
- Confidence: High

## Fix Priority
1. 立即要修
   - 停止在前端寫入/持久化玩家 token，改只信任後端 `HttpOnly` cookie
   - 封住 `/games/submissions` 的可腳本化升級路徑
   - 把 admin session 改成獨立短期 session，而不是直接重用 `ADMIN_KEY`
   - 移除 URL token 登入，至少先在前端立即清掉 `?token=`
2. 可不修（因為系統只活 1 天）
   - `/metrics` 或 `/docs` 的外部可見性，若實際上已被內網/Ingress 擋住
   - dev compose 內的預設帳密，只要確定不會對外暴露
