# Security Review Report
- Project: 2026-game
- Stack: Go chi API + Next.js
- Context: system lifetime = 1 day
- Reviewed at: 2026-03-23_16-59-05

## Executive Summary
- 目前最值得先修的是：repo 內直接提交了 `ADMIN_KEY`/DB 密碼、可直接登入的使用者 token 與實際登入連結，以及前端把 token 放進 URL / localStorage / JS 可讀 cookie。
- 這些不是長期理論風險；只要拿到 repo、共享裝置、瀏覽器歷史紀錄或一個外洩連結，1 天內就足以造成管理員接管、玩家帳號接管、折價券濫用與活動流程被操控。
- 另外，登入端點缺少登入前節流，會放大前述弱 token / 已外洩 token 的利用效率。
- 其餘像公開 `/metrics`、`/docs` 與目前 `next` 版本漏洞，屬於次優先，但修補成本也低。

## Findings
### [Critical] Repo committed live admin/database secrets
- Where: `backend/.env`, `backend/pkg/config/config.go`
- Why it matters: 若這些值仍在部署環境使用，攻擊者可直接取得 admin 權限與資料庫存取；對 1 天系統仍是立即可用的接管路徑。
- Evidence: `backend/.env:1` 含 `ADMIN_KEY=...`，`backend/.env:7`-`backend/.env:12` 含實際 DB 連線資訊；`backend/pkg/config/config.go:8` 自動載入 `.env`；`backend/internal/handler/admin/login.go:38` 與 `backend/pkg/middleware/admin_key.go:20` 直接以 `ADMIN_KEY` 做登入與授權比對。
- Fix: 立刻輪替 `ADMIN_KEY` 與 DB 密碼；把 `backend/.env` 從 repo 與建置流程移除；admin 改成獨立 session secret，不要重用登入金鑰。
- Confidence: High

### [Critical] Predictable committed login links allow direct player account takeover
- Where: `backend/data/SITCON_2026_users.csv`, `backend/scripts/generate_users_from_csv/main.go`, `backend/internal/handler/user/login.go`, `frontend/app/login/page.tsx`
- Why it matters: 若這批 seed 資料被匯入正式環境，任何人知道或猜到 `1000`, `1001`... 就能直接登入他人帳號，改分數、拿 coupon、看個資。
- Evidence: `backend/data/SITCON_2026_users.csv:1`-`backend/data/SITCON_2026_users.csv:40` 直接存放 `https://2026-game.sitcon.party/login?token=1000` 這類登入連結；`backend/scripts/generate_users_from_csv/main.go:106` 把 CSV 的 `Code` 原樣寫成 `auth_token`；`backend/internal/handler/user/login.go:53` 先用 token 查 DB，查到即登入，不再向 OPass 驗證；`frontend/app/login/page.tsx:101`-`frontend/app/login/page.tsx:115` 會自動用 URL 裡的 `token` 登入。
- Fix: 若正式環境用了這批 seed，立即全部失效並重發不可預測 token；登入 token 不要出現在 query string；對匯入資料加註用途並隔離正式/測試資料。
- Confidence: High

### [High] Staff/booth/admin secrets are accepted via query string and persisted client-side
- Where: `frontend/app/(internal)/admin/page.tsx`, `frontend/app/(internal)/cashier/page.tsx`, `frontend/app/(internal)/booth/page.tsx`, `frontend/stores/staffStore.ts`, `frontend/stores/boothStore.ts`, `backend/data/staffs.json`, `backend/data/activities.json`
- Why it matters: URL token 很容易進入瀏覽器歷史、截圖、分享訊息與現場共用裝置；一旦外洩即可直接操作 admin / cashier / booth 流程。
- Evidence: `frontend/app/(internal)/admin/page.tsx:13`、`frontend/app/(internal)/cashier/page.tsx:85`、`frontend/app/(internal)/booth/page.tsx:27` 都從 `?token=` 自動登入；`frontend/stores/staffStore.ts:12`、`frontend/stores/boothStore.ts:12` 用 Zustand `persist` 落地到 localStorage；repo 內也直接存放 staff/booth token 於 `backend/data/staffs.json:5` 與 `backend/data/activities.json:4`。
- Fix: 改成一次性登入連結或 POST body/QR 掃碼交付，不要用 query string；staff/booth token 不要持久化到 localStorage；若 seed token 曾上線，立即輪替。
- Confidence: High

### [High] Frontend defeats HttpOnly session protection for player auth
- Where: `frontend/stores/userStore.ts`, `frontend/hooks/api/useUser.ts`, `backend/pkg/helpers/cookie.go`
- Why it matters: 後端本來有設定 `HttpOnly` cookie，但前端又把同一個 bearer token 存進 localStorage 與 JS 可讀 cookie；任何 XSS、惡意 extension、共用裝置都能直接偷走登入憑證。
- Evidence: `backend/pkg/helpers/cookie.go:19` 將 server cookie 設為 `HttpOnly`; 但 `frontend/hooks/api/useUser.ts:59`-`frontend/hooks/api/useUser.ts:61` 在登入成功後持久化 raw token；`frontend/stores/userStore.ts:14` 使用 `persist`，且 `frontend/stores/userStore.ts:21` 以 `document.cookie = token=...` 建立非 HttpOnly cookie。
- Fix: 前端不要保存 raw bearer token；只依賴後端 `HttpOnly` session cookie；清除既有 `localStorage`/JS cookie token 寫入邏輯。
- Confidence: High

### [Medium] No pre-auth rate limit on login endpoints
- Where: `backend/pkg/middleware/session_rate_limit.go`, `backend/internal/router/user.go`, `backend/internal/router/admin.go`, `backend/internal/router/discount.go`, `backend/internal/router/activities.go`
- Why it matters: 現有 rate limit 只保護「已登入」請求；登入前可以無限制嘗試 token，會放大弱 token、外洩 token 或撞號的成功率。
- Evidence: `backend/pkg/middleware/session_rate_limit.go:28`-`backend/pkg/middleware/session_rate_limit.go:45` 只接受已放入 context 的 user/staff/booth/admin 當 key；`/users/session`、`/admin/session`、`/discount-coupons/staff/session`、`/activities/booth/session` 在各 router 中都未套 pre-auth limiter。
- Fix: 對所有 session 建立端點加上 IP/base token 維度的 pre-auth rate limit；登入失敗統一回應並記錄告警。
- Confidence: High

### [Medium] Public recon surfaces are enabled in prod settings
- Where: `backend/cmd/api/main.go`, `backend/.env`
- Why it matters: 雖然不一定直接打穿，但對 1 天攻防來說，公開 API 文件與 metrics 會顯著降低攻擊者摸索成本。
- Evidence: `backend/cmd/api/main.go:145` 無條件掛載 `/metrics`；`backend/cmd/api/main.go:161` 在 `AppDocs` 開啟時暴露 `/docs`；`backend/.env:3` 設 `APP_DOCS=true` 且 `backend/.env:4` 是 `APP_ENV=prod`。
- Fix: 正式環境關閉 `/docs`，將 `/metrics` 僅限內網/監控來源存取，或加上反向代理 ACL。
- Confidence: High

### [Medium] Self-hosted Next.js version has a published rewrite smuggling issue
- Where: `frontend/package.json`, `frontend/next.config.ts`, `frontend/Dockerfile`
- Why it matters: 目前前端是自架 `next`，且使用 external rewrite 代理 `/api`；命中官方已公告的 request smuggling 漏洞時，可能繞過對後端路徑的假設。
- Evidence: `frontend/package.json:18` 使用 `next 16.1.6`; `frontend/next.config.ts:5`-`frontend/next.config.ts:11` 設定 rewrite 代理 `/api/:path*`; `pnpm audit --prod` 回報 `GHSA-ggv3-7p47-pfv8` / `CVE-2026-29057`，修補版為 `16.1.7`。
- Fix: 升級到 `next >= 16.1.7`；若暫時不能升級，先在 edge/proxy 阻擋 rewritten routes 的 chunked `DELETE`/`OPTIONS`。
- Confidence: Medium

## Fix Priority
1. 立即要修
   - 輪替並移除 repo 內的 `ADMIN_KEY`/DB 密碼
   - 失效所有已提交或可預測的 user/staff/booth token
   - 移除 URL token 登入與 localStorage/JS cookie 持久化
   - 對所有 login/session 端點加 pre-auth rate limit
   - 關閉 prod `/docs`，限制 `/metrics`
   - 升級 `next` 到 `16.1.7+`
2. 可不修（因為系統只活 1 天）
   - 報告中未列出的純防禦深度項目；目前沒有比上述更值得搶修的低成本高價值問題
